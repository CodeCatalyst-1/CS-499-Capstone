Professional Self-Assessment

Introduction

Completing the Computer Science program at Southern New Hampshire University and developing this ePortfolio has been a process of consolidating skills that were initially learned in isolation into a coherent professional identity. Early coursework introduced individual topics such as data structures, database design, and software testing as separate disciplines. The capstone experience required integrating those disciplines into a single artifact and reflecting on how they connect to one another and to the broader demands of the field.

My professional goal is to work in data science and analytics, a specialization that sits at the intersection of software engineering, database management, and algorithmic thinking. The Grazioso Salvare Animal Shelter Dashboard, which serves as the artifact for all three enhancement categories, is a data-driven application that queries a MongoDB database, applies filtering and aggregation logic, and presents interactive visualizations to support decision-making. Building and refining this system across three milestones gave me a concrete vehicle for demonstrating the skills that a data science role demands.

Collaborating in a Team Environment

Several courses required collaboration that shaped how I approach team-based development. In CS 250 (Software Development Lifecycle), I worked within an Agile framework that emphasized iterative feedback, sprint planning, and aligning technical work with stakeholder needs. The code review at the beginning of this capstone reinforced that lesson — walking through my own code as if presenting to a peer required me to articulate design decisions clearly and identify weaknesses honestly. The capstone also required using GitHub as a version control and portfolio platform, a core collaboration tool in professional software development.

Communicating with Stakeholders

The narrative documents written for each milestone required translating implementation details into language that explains not just what was changed but why the change matters. The dashboard itself is a form of stakeholder communication — designed for staff who need to identify animals suitable for rescue training. The interface presents complex query results through a data table, pie chart, bar chart, and geolocation map, each chosen to answer a specific question the user is trying to ask.

Data Structures and Algorithms

The original dashboard used identical placeholder filters for all three rescue categories. The enhancement replaced this with a RESCUE_CRITERIA data structure encoding each category as a specification object with breed lists, sex requirements, and age bounds. The apply_rescue_filter() function composes a compound boolean mask from these specifications. This demonstrated that the choice of data structure directly affects the clarity and correctness of the algorithm that operates on it.

Software Engineering and Database

The software engineering enhancement connected the dashboard to MongoDB through the CRUD module, replaced print statements with logging, added type hints and input validation, and extracted filter definitions into a standalone function. The database enhancement added indexing, aggregation pipelines, bulk import with duplicate handling, and distinct value queries — transforming MongoDB from a flat file store into a properly utilized database engine.

Security

Security considerations were addressed at multiple levels: credentials encoded with quote_plus before URI embedding, argument validation on every public CRUD method, and an empty-query guard on delete() to prevent accidental collection wipes. These reflect the kind of defensive thinking that prevents common categories of data loss in real-world applications.

Milestone One

Code Review

The code review analyzed the original Grazioso Salvare Dashboard artifact from CS 340 across three areas required by the capstone: a walkthrough of existing functionality, a code analysis targeting structure, logic, efficiency, security, testing, and documentation, and an enhancement plan mapping planned improvements to course outcomes across all three categories.

The analysis identified several critical issues: the dashboard never used the CRUD module (loading data from CSV instead), all three rescue filters returned identical results due to placeholder logic, error handling relied on bare print() statements, and there were no input validation guards on any CRUD method. The enhancement plan outlined three categories of improvement — software design and engineering, algorithms and data structures, and databases — each mapped to specific course outcomes.

Milestone Two

Enhancement One: Software Design & Engineering

The most significant defect in the original code was that all three rescue-type filters used identical conditions with placeholder comments. The enhanced version implements the official Grazioso Salvare breed, sex, and age-range criteria for each rescue type. The dashboard was also reconnected to MongoDB through the CRUD module — the original code loaded data from a CSV and never used crud.py at all.

Additional improvements included extracting filter definitions into a standalone build_filter_map() function, replacing print() with Python's logging module, adding type hints and input validation to all CRUD methods, and replacing the deprecated ismaster command with ping.

Original crud.py
class AnimalShelter:
    def __init__(self, user, passwd, ...):
        self.client = MongoClient(uri)
        self.client.admin.command('ismaster')
        print(f" Connected to MongoDB...")

    def read(self, query):
        if not self.client:
            return []
        try:
            return list(self.collection.find(query))
        except Exception as e:
            print(f" Read failed: {e}")
            return []
Enhanced crud.py
class AnimalShelter:
    def __init__(self, user: str, passwd: str, ...) -> None:
        self.client = MongoClient(uri, serverSelectionTimeoutMS=5000)
        self.client.admin.command("ismaster")
        logger.info("Connected to MongoDB as user '%s'.", user)
        self._ensure_indexes()

    def read(self, query: dict, projection: dict | None = None) -> list:
        if not isinstance(query, dict):
            logger.error("read() requires a dict query.")
            return []
        cursor = self.collection.find(query, projection)
        return list(cursor)
Milestone Three

Enhancement Two: Algorithms & Data Structures

The original filter_map pre-computed filtered DataFrames at module load time, and all three rescue categories returned identical results — thousands of undifferentiated dog records. The enhancement introduced a RESCUE_CRITERIA data structure that encodes each rescue category as a specification object, and an apply_rescue_filter() function that composes a compound boolean mask from those specifications.

Validation against the AAC dataset confirmed meaningfully distinct results: Water Rescue returns 17 records, Mountain or Wilderness Rescue returns 11, and Disaster Rescue or Individual Tracking returns 17. The filtering algorithm became self-documenting — each condition in the boolean mask directly mirrors the specification it encodes.

Milestone Four

Enhancement Three: Databases

The original implementation treated MongoDB as a flat file store — no indexes, no aggregation, no bulk operations. The enhancement added five single-field indexes and two compound indexes that mirror the dashboard's rescue filter queries. An aggregate_outcomes() method runs a $match → $group → $sort pipeline inside the database engine, and a bulk_import() utility handles batch insertion with duplicate resilience via ordered=False.

On the dashboard side, filter definitions were rewritten as MongoDB query dictionaries passed directly to shelter.read() and shelter.aggregate_outcomes(), moving filtering from Python into the database layer. A breed distribution bar chart was also added alongside the existing pie chart and geolocation map.

Indexing

Automatic Index Management

Five single-field and two compound indexes created at startup with background=True for non-blocking builds. Idempotent — safe to call on every restart.

Aggregation

Pipeline Queries

aggregate_outcomes() computes grouped counts inside MongoDB, sending only summary rows over the network instead of all raw documents.

Bulk Operations

Resilient Import

bulk_import() with ordered=False continues past duplicates and reports inserted vs. skipped counts from the BulkWriteError details.
